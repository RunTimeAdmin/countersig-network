Countersig Network is an open protocol that gives AI agents verifiable on-chain identities, cryptoeconomically-secured reputation scores, and trustless agent-to-agent authentication — without a central authority.

Register an Agent View on GitHub

The Problem

AI agents can execute tasks, transact on-chain, and communicate via A2A protocols. But they cannot prove who they are. Without verifiable identity, every agent interaction requires pre-existing trust — which does not scale.

🎭

No Verifiable Identity

Any agent can claim to be any other agent. There is no cryptographic proof of ownership.

📊

No Portable Reputation

Reputation is siloed per platform. An agent starts from zero on every new marketplace.

⚠️

No Accountability

When an agent causes harm, there is no stake to slash and no operator to hold responsible.



The Protocol

Three on-chain registries, one staking core, and a TypeScript SDK.

LAYER 3

Off-Chain Verification

Agents authenticate peer-to-peer using Ed25519 PKI challenge-response. On-chain registries serve as the source of truth for public keys and reputation scores.

LAYER 2

Decentralized Identity Layer

The did:countersig method resolves on-chain state to W3C-compliant DID Documents. Any resolver can reproduce the DID hash without querying the contract.

LAYER 1

On-Chain State (EVM)

Three UUPS-upgradeable Solidity registries: Identity (DID anchoring), Reputation (6-factor score), and Staking (bonding + slashing). All state is public and composable.

How It Works

From zero to a live, reputation-tracked AI agent in five steps.

1

Generate an Ed25519 keypair

Your agent's cryptographic identity lives off-chain. The public key gets registered on-chain; the private key signs challenges when peers ask your agent to prove who it is. No central authority issues or controls it.

2

Stake $CSIG and register your DID

Deposit a minimum $CSIG bond, then call registerAgent(agentAddress, ed25519PubKey). The contract computes your didHash from the chain ID and address, stores the public key, and emits an event. Your DID — did:countersig:<chainId>:<address> — is now globally resolvable. The stake creates skin in the game: misbehavior leads to slashing and permanent loss of the bond.

3

Do work — with your DID on every action

Include agent_did in every audit call to CounterAudit. Each action is sealed into a tamper-evident packet. CounterAudit queries your live on-chain identity and reputation score at seal time and embeds both inside the AES-GCM-sealed, RFC 3161-timestamped record. The reputation score is frozen at the moment of the action — not retroactively updated.

4

The oracle computes your reputation

An oracle network watches on-chain signals — fee volume, task attestations, registration age, external trust sources — and writes a 6-factor score (0–100) to the CountersigReputation contract every epoch. Any smart contract or off-chain consumer can call getTotalScore(didHash) or meetsThreshold(didHash, minScore). The computation is off-chain; the result is on-chain.

5

Reputation unlocks trust — and more work

Peers that require proof of identity issue an Ed25519 challenge. Your agent signs it. The verifier resolves your public key from the contract and checks the signature without contacting any third party. Higher reputation means peers set lower barriers — your agent gets access to higher-value work, which generates more fees, which increases your reputation further. The loop is self-reinforcing and Sybil-resistant by design.

Why the sealed packet matters

Each CounterAudit packet freezes the agent's reputation at the moment of the action. If an agent is slashed next month, every audit record from before the slash still shows what their reputation was then. The historical record does not rewrite itself — the RFC 3161 timestamp and AES-GCM seal together guarantee it.

DID Method Specification

The did:countersig method assigns each AI agent a globally unique, on-chain identifier.

did:countersig:<chainId>:<agentAddress>
ComponentDescription
did:countersigMethod prefix. Identifies this as a Countersig Network DID.
<chainId>EVM chain ID (e.g. 1 for mainnet, 11155111 for Sepolia).
<agentAddress>The agent's Ethereum address. The operator registers the agent's Ed25519 public key against it.

Register an Agent — SDK Quickstart

// npm install @countersig/protocol-sdk
import { CountersigAgent, registerAgent } from '@countersig/protocol-sdk';
import { ethers } from 'ethers';

// Generate a fresh Ed25519 keypair — no external crypto imports needed.
const { agent, privateKey } = CountersigAgent.generate({
  agentAddress: '0xYourAgentAddress',
  chainId:      11155111,
});
// Store privateKey (0x hex) securely — it cannot be recovered from the agent.

console.log(agent.did);
// → did:countersig:11155111:0xYourAgentAddress

const { didHash, txHash } = await registerAgent(
  operatorWallet, agent.agentAddress, agent.publicKeyBytes32, IDENTITY_ADDRESS
);

Agent-to-Agent Authentication

import { CountersigVerifier } from '@countersig/protocol-sdk';

const verifier = new CountersigVerifier({ rpcUrl, addresses, chainId: 11155111 });

// B challenges A, A signs, B verifies against on-chain public key
const challenge = agentB.issueChallenge(agentA.did);
const signature = agentA.signChallenge(challenge.payload);
const valid     = await verifier.verifySignature(agentA.did, challenge.payload, signature);
const trusted   = await verifier.meetsThreshold(agentA.did, 60);
